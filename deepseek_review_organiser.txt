## 1. Database Schema and Security

### Table Design
- The table `organiser_applications` has appropriate columns and constraints.
- The use of a CHECK constraint for `status` is good.
- The `league_url` is stored as TEXT, which is acceptable given the plan to validate it.

### Indexes
- Indexes are proposed for status (for admin queries) and user_id (for user's own applications). These are appropriate.

### RLS Policies
- The plan includes RLS policies. However, note that the plan does not specify if RLS is enabled on the table. We must ensure RLS is enabled.
- The policies described are:
    - Users can SELECT their own applications.
    - Users can INSERT their own applications.
    - Users can UPDATE their own applications only to set status = 'withdrawn' (and only when current status is 'pending').
    - Admins can SELECT and UPDATE all applications.

  We must ensure that the UPDATE policy for users is correctly written to only allow updating the status to 'withdrawn' and only when the current status is 'pending'. The plan does not provide the exact SQL for the policies, so we must be cautious.

  Example policy for user update (using Supabase RLS):

  ```sql
  CREATE POLICY "Users can withdraw their own pending applications"
  ON public.organiser_applications
  FOR UPDATE
  USING (auth.uid() = user_id AND status = 'pending')
  WITH CHECK (auth.uid() = user_id AND status = 'withdrawn');
  ```

  However, note that the above policy would allow updating only to 'withdrawn'. But note that the plan says "Users can UPDATE their own applications only to set status = 'withdrawn' (where current status = 'pending')". This is a bit tricky because we want to allow the update only if the new status is 'withdrawn' and the old status is 'pending'. The above policy uses `USING` to check the old row and `WITH CHECK` for the new row. But note that in Supabase, the `WITH CHECK` expression is evaluated on the new row. So the above policy would require that the new row has status 'withdrawn' and the user_id matches. Also, the `USING` clause ensures that the old row has status 'pending' and the user_id matches.

  Alternatively, we might want to use a trigger to ensure that the only update allowed by the user is to set status to 'withdrawn' and only from 'pending'. However, the plan says to use RLS. We can do it with RLS as above, but note that the user might try to update other columns. The policy above would allow updating any column as long as the conditions are met. To be strict, we might want to ensure that only the status can be updated (and maybe `updated_at`). But the plan does not specify. We could also add a trigger for more complex logic.

### Constraint: One pending application per user
- The partial unique index for one pending application per user is a good idea to prevent multiple pending applications.

### Potential Risks and Edge Cases:
- **RLS on INSERT**: The plan says users can INSERT their own applications. We must ensure that the user_id in the inserted row is the same as the authenticated user. This can be enforced by setting the user_id from the server action (using `auth.uid()`) or by a policy that forces the user_id to be the current user. The plan does not specify how the user_id is set. It is safer to set it in the server action (from the session) rather than relying on the client to send it.

- **Admin Policies**: The plan says admins can SELECT and UPDATE all applications. We must have a way to identify admins. This is typically done by checking the user's role in the `profiles` table. We must ensure that the admin policy is correctly written to only allow users with an admin role. The plan does not specify the exact policy condition.

- **Data Integrity**: The `reviewed_by` column references `profiles(id)`. We must ensure that when an admin reviews, this is set. Also, note that the `admin_notes` is optional.

- **Cascading Deletes**: The table uses `ON DELETE CASCADE` for `user_id`. This is acceptable because if a user is deleted, their applications should be deleted too.

11. **Update Timestamps**: The table has `created_at` and `updated_at`. We should set up a trigger to update `updated_at` automatically, or use the `updated_at` column with a default of `now()` but note that `updated_at` must be updated on change. The plan does not specify a trigger for `updated_at`. We can use the following:

    ```sql
    CREATE TRIGGER update_organiser_applications_updated_at
    BEFORE UPDATE ON public.organiser_applications

    ```

    Assuming the function `update_updated_at_column()` exists (commonly used in Supabase projects).

12. **Indexes**: The proposed indexes are good, but also consider an index on `reviewed_by` if we plan to query by admin reviewer often.

CREATE POLICY "Users insert own applications" 
ON organiser_applications FOR INSERT 
WITH CHECK (auth.uid() = user_id);

CREATE POLICY "Users withdraw own pending applications" 
ON organiser_applications FOR UPDATE 
USING (auth.uid() = user_id AND status = 'pending')
WITH CHECK (auth.uid() = user_id AND status = 'withdrawn');

  EXISTS (SELECT 1 FROM profiles WHERE id = auth.uid() AND role IN ('admin', 'super_admin'))

### 2. **Race Condition in Role Promotion**
**Risk**: Medium  
**Issue**: Server action `reviewOrganiserApplication` calls `updateUserRole(userId, 'organizer')` directly. If the user already has organizer access or if there are concurrent approvals, this could cause issues.

**Recommendation**:

  .from('profiles')

  await supabase
    .from('profiles')

### 3. **URL Validation Bypass Risk**
**Risk**: Medium  
**Issue**: Checking only for `pokemon.com` in hostname could be bypassed with:
- `pokemon.com.evildomain.com`
- Unicode homograph attacks
- URL shorteners pointing to valid domains

**Recommendation**:
```typescript
// Use URL API and strict domain checking
function validatePokemonLeagueUrl(url: string): boolean {
  try {
    const parsed = new URL(url);
    const hostname = parsed.hostname.toLowerCase();
    
    // Exact domain matching or subdomains of pokemon.com
    return hostname === 'pokemon.com' || 
           hostname === 'www.pokemon.com' ||
           hostname.endsWith('.pokemon.com');
  } catch {
    return false;
  }
}
```
