 Catastrophic Edge Case 1: RLS Bypass via Service Role in Trigger Functions

    - The function might not properly check the user_id of the inserted row, and instead rely on the RLS of the base table, but since it's running as service role, it bypasses RLS.

Alternatively, we can design the system so that the trigger function does not need to bypass RLS.

However, note that in a trigger, we don't have the current session context if the trigger is fired by an insert from the service role. Therefore, we must ensure that the application never uses the service role for user operations. Instead, use a separate role that is subject to RLS for user operations, and only use the service role for admin operations.

    But note: The application might use the service role for cron jobs and webhooks. We must be cautious.

    Concrete fix:
    - For user-triggered operations (like registering for a tournament), use a database role that is subject to RLS. Do not use the service role for these operations.
    - For trigger functions that need to check capacity, do not rely on the user's session. Instead, the trigger function should check the capacity by querying the queue table and the tournament settings. Since the trigger is running in the same transaction as the insert, it can see the new row and the existing rows. We can use a BEFORE INSERT trigger that aborts if the capacity is exceeded.

    Example of a trigger function for capacity check:

    CREATE OR REPLACE FUNCTION check_capacity()
    RETURNS TRIGGER AS $$
    DECLARE
        current_global_count integer;
        current_division_count integer;
        tournament_capacity integer;
        division_capacity integer;
    BEGIN
        -- Get the tournament and division capacity
        SELECT global_capacity, division_capacity
        INTO tournament_capacity, division_capacity
        FROM tournaments
        WHERE id = NEW.tournament_id;

        -- Count current global and division queue entries (including the one being inserted)
        SELECT COUNT(*) INTO current_global_count
        FROM queue_entries
        WHERE tournament_id = NEW.tournament_id;

        SELECT COUNT(*) INTO current_division_count
        FROM queue_entries
        WHERE tournament_id = NEW.tournament_id AND division = NEW.division;

        -- Check global capacity
        IF tournament_capacity IS NOT NULL AND current_global_count > tournament_capacity THEN
            RAISE EXCEPTION 'Global capacity exceeded for tournament %', NEW.tournament_id;
        END IF;

        -- Check division capacity
        IF division_capacity IS NOT NULL AND current_division_count > division_capacity THEN
            RAISE EXCEPTION 'Division capacity exceeded for tournament %, division %', NEW.tournament_id, NEW.division;
        END IF;

        RETURN NEW;
    END;
    $$ LANGUAGE plpgsql;

1. **RLS Policy Bypass During Concurrent Registration Stampede**  
   **Doomsday Scenario:** When thousands of users simultaneously register for a popular tournament, Supabase RLS policies that rely on `auth.uid()` may experience catastrophic latency spikes. If the authentication service slows even slightly, RLS policies could temporarily evaluate `auth.uid()` as `NULL` for legitimate users due to timeouts. This opens a window where a malicious actor sends crafted requests with delayed authentication tokens, causing RLS to incorrectly allow or deny operations. Worse, if RLS policies have conditional checks that depend on subqueries (e.g., checking if a user is already in the queue), the concurrent load could cause those subqueries to return stale data, allowing duplicate registrations or overfilling capacity.

   **Technical Solution:** Implement a mandatory two‑tier validation layer. First, use PostgreSQL’s `pg_notify` to push registration attempts through a dedicated, single‑threaded queue processor written in PL/pgSQL or a robust external service (e.g., Elixir/OTP process) that holds a global mutex. This processor validates the user’s authentication state via a direct lookup to the `auth.users` table (bypassing RLS but with strict service‑role trust boundaries) before applying business rules. Second, wrap all RLS policies with `CHECK (true)` and rely solely on this processor for writes; convert RLS to read‑only for users. This decouples authentication latency from write‑time policy evaluation.

2. **Webhook Idempotency Key Collision Under High Concurrency**  
   **Doomsday Scenario:** Your system uses payment‑provider webhooks (Stripe) to confirm registrations. You’ve implemented idempotency keys stored in a `processed_webhooks` table with a `UNIQUE` constraint. However, during a registration surge, two separate webhooks for the same payment event might be delivered microseconds apart. The first webhook acquires an advisory lock, processes, and commits. The second webhook, already in‑flight, hits the database during the same `READ COMMITTED` transaction isolation window before the first transaction’s commit is visible. The unique constraint check sees no duplicate yet, so both threads proceed to increment capacity counters, leading to double‑counting and overbooking.

   **Technical Solution:** Use `SERIALIZABLE` isolation level for all webhook‑processing transactions combined with explicit row‑level locking on the tournament capacity row before any other operations. Additionally, use a composite idempotency key of `(webhook_id, tournament_id, player_id)` and perform an `UPSERT` with `ON CONFLICT DO NOTHING` at the very start of the transaction, before any business logic. This ensures that even under serializable isolation, the second webhook will block until the first commits and then immediately abort due to the conflict. Pair this with a circuit‑breaker pattern that rejects webhooks after a threshold of retries to prevent third‑provider retaliation.
