## Reasoning
We are dealing with a high-stakes registration system where even small oversights can lead to unfairness, data corruption, or system failure during a stampede. Let's think beyond the already solved issues.

Edge Case 1: **Malicious Player Exploiting the Pending Payment Window by Creating Multiple Queued Entries**

- **Scenario**: A player uses multiple emails or accounts to join the queue and get multiple slots in the pending payment state. They then intentionally let some expire, hoping to free up capacity for their other entries (which are further back in the queue) or to resell slots.
- **Doomsday**: This can lead to an unfair distribution of slots, where one player can hold multiple slots hostage and then release them at strategic times (e.g., to friends) or cause unnecessary churn in the queue, delaying legitimate players.
- **Solution**: Implement a strict one-slot-per-player (or per team) policy across the event. This can be enforced by checking against a unique identifier (like user ID or team ID) when joining the queue and during the pending payment window. Additionally, consider tying the pending payment to a specific queue entry and not allowing the same player to have multiple pending payments. If a player fails to pay for their first slot, they cannot get another until the first one expires and is cleaned up.

Edge Case 2: **Organizer Adjusts Capacity or Divisions While Queue is Active, Causing Inconsistencies**

- **Scenario**: The organizer reduces the total capacity or a division's capacity while there are already players registered, in the queue, or in pending payment. This could lead to overcapacity in a division or globally, or cause players to be unfairly bumped.
- **Doomsday**: For example, if global capacity is reduced from 100 to 50 and there are already 80 registered, then 30 players must be removed. But which 30? If done arbitrarily, it could be unfair and break the queue order.
- **Solution**: When capacity is adjusted (especially reduced), the system must recalculate the current state and adjust accordingly. This should be done in a transaction with the same advisory locks used during registration. The algorithm should:
  - First, adjust the divisions proportionally (if divisional capacity changes) and then globally.
  - For any overcapacity, start by removing players who are in the queue (not yet registered) in reverse order of queue position (last in, first out). Then, if still over, remove from the pending payment group (again, reverse order of entry). Finally, if still over, remove from the registered group (but this is extreme and should be avoided). Alternatively, the organizer could be forced to only increase capacity or to wait until the queue is empty to decrease. However, a safer approach is to freeze the queue when capacity changes and require the organizer to manually resolve overcapacity (e.g., by moving players to a waitlist or another division) before reactivating.

Edge Case 3: **Player Changes Division While in Queue, Leading to Queue Position Gaming**

- **Scenario**: A player joins the queue for a division that has a shorter queue (or more capacity) and then switches to a more competitive division after securing a better position, effectively jumping the queue in the target division.
- **Doomsday**: This undermines the fairness of the queue system, as players can exploit division switches to get ahead.
- **Solution**: If a player is allowed to change divisions while in the queue, their queue position should be reset to the end of the queue for the new division. Additionally, the system should check that the player is not allowed to switch to a division that is already at capacity (or over). The switch should be treated as leaving one queue and joining another, so the player loses their original spot. This must be communicated clearly to the player before they confirm the switch.

Edge Case 4: **Time Drift on Client Devices Causing Early or Late Payment Submission**

- **Scenario**: The client's clock is out of sync with the server, causing the 10-minute payment countdown to appear shorter or longer than it actually is. A player might be unfairly penalized because their local clock was slow and they missed the payment window, or they might submit payment early (which is less of an issue).
- **Doomsday**: Players lose their slot due to client-side time drift, leading to support complaints and unfairness.
- **Solution**: Never rely on the client's clock for critical timing. Use server timestamps for all deadlines. The client should display the countdown based on the server's time (by sending the deadline timestamp and having the client calculate the remaining time until that absolute time, but also periodically sync with the server to adjust for client clock drift). Alternatively, the client can poll the server for the remaining time (which is less efficient but more accurate). In the payment window, the server must reject payments that are after the deadline (based on server time).

Edge Case 5: **Database Performance During Queue Position Calculation for 50,000+ Players**

- **Scenario**: Every player in the queue is polling every 5 seconds to get their position. The database query to calculate the position (which might require counting how many players are ahead of them in the queue) becomes very expensive and slows down the entire system.
- **Doomsday**: The database becomes a bottleneck, leading to slow responses, timeouts, and a poor user experience. It might even crash during the stampede.
- **Solution**: Instead of calculating the position on the fly for each poll, consider caching the queue order in Redis or using a materialized view that is updated periodically (e.g., every 10 seconds). Alternatively, use a queue management system that assigns a sequential number to each queued player and then uses that number to determine position. This number can be stored in the database and indexed. When a player is removed (due to payment, expiration, etc.), the numbers don't change, but we can recalculate periodically (or on demand) to fill gaps. However, note that the queue order might change due to priority or other business rules, so the sequential number must be assigned at join time and then adjusted if needed (which is complex). Another approach is to use a row number in a view, but that can be expensive. We can also consider partitioning the queue by division and then using a counter per division. The key is to avoid a full table scan or count for each poll.

Additional Consideration: **Supabase Connection Limits and Vercel Timeouts**

- **Scenario**: During the stampede, thousands of clients are polling every 5 seconds, leading to a high number of concurrent database connections and function invocations. This might exceed Supabase's connection limits or cause Vercel function timeouts (if we are using serverless functions to handle the polling).
- **Doomsday**: The system becomes unresponsive, and players cannot join the queue or see their status.
- **Solution**: Use a combination of:
  - Client-side adaptive polling (which we already have) to reduce the load when the queue is not moving.
  - Consider using Supabase's real-time capabilities (but we avoided WebSockets to prevent storming). However, for the queue position, we might use a publish-subscribe model where the server broadcasts the queue state periodically (e.g., every 10 seconds) to all connected clients, reducing the number of individual requests.
  - For Vercel, ensure that the functions are optimized and have a short execution time. Use connection pooling (with PgBouncer) for database connections. Also, consider using a dedicated server (or a longer timeout) for the queue status endpoint if necessary.

These edge cases require careful design and testing. The solutions proposed aim to maintain fairness, performance, and reliability under extreme conditions.

## Output
1. **Malicious Slot Squatting with Distributed Bots**  
**Edge Case**: An attacker deploys bots to claim every queue slot using burner accounts, then intentionally abandons the pending_payment window at staggered intervals. This creates artificial churn, forcing legitimate players to constantly re-poll and potentially miss the 10-minute window due to UI fatigue, while the attacker's real account waits to snipe a slot during the chaos.  
**Doomsday Scenario**: Legitimate players never secure slots despite constant monitoring, causing community outrage and rendering the queue system unfair.  
**Technical Solution**: Implement a probabilistic bot-detection layer at queue entry using invisible challenges (e.g., hashcash-like proof-of-work tokens) and require a verified account history (previous tournaments, attached gaming profiles). Additionally, silently shadow-ban suspicious actors by letting them proceed normally but placing them in a isolated queue pool that doesn't impact real capacity.

2. **Mid-Event Capacity Reduction Causing Transactional Deadlock**  
**Edge Case**: An organizer decreases global or divisional capacity while the queue is live, but the advisory-locked transaction is already processing a batch of promotions from queue to registered. This creates a race where the system must decide whether to abort mid-transaction or allow temporary overcapacity.  
**Doomsday Scenario**: The deadlock forces a rollback, stranding hundreds of players in a "zombie state" where they lose their queue position but aren't fully registered, requiring manual database intervention.  
**Technical Solution**: Wrap capacity updates in the same advisory lock used for promotions, and implement a two-phase capacity change: first, freeze all queue movements and drain pending operations; then recalculate positions and adjust states atomically. Add an admin dashboard warning showing how many active registrations will be affected before allowing the reduction.

3. **Cross-Device Payment Window Exploitation**  
**Edge Case**: A player joins the queue on a device with a deliberately skewed system clock, then switches to a second device with accurate time to complete payment. The client-side countdown timer becomes desynchronized, causing the player to believe they have more (or less) time than allotted, leading to premature abandonment or late payment attempts that overwhelm the payment webhook handler.  
**Doomsday Scenario**: Payment webhooks arrive outside the valid server-side window, triggering erroneous cancellations or allowing expired slots to be paid for, corrupting capacity counts.  
**Technical Solution**: Never trust client timestamps. Use server-sent countdown timers via Server-Sent Events (SSE) or a secure API that returns remaining seconds based on the server's authoritative clock. Invalidate payment links after the server-side window expires, and implement idempotent webhook handling that rejects payments for already-cancelled slots.

4. **Division Switching Queue-Jumping via Concurrent Requests**  
**Edge Case**: A player in the "Juniors" queue (shorter line) rapidly switches divisions to "Masters" while simultaneously confirming the switch from another browser tab, exploiting a race condition where the system releases their old slot after acquiring the new one, effectively holding two positions.  
**Doomsday Scenario**: The player can cherry-pick the division that opens up first, gaming the system and causing fairness violations detectable only via audit logs.  
**Technical Solution**: Implement a per-player mutex (e.g., Redis lock with player ID as key) for all queue state mutations, including division switches. Enforce a strict state machine where the old queue position is released atomically only after the new division's queue entry is created, and add a cooldown period (e.g., 30 seconds) before allowing another division change.

5. **Supabase Connection Pool Exhaustion from Cascading Polling**  
**Edge Case**: When the queue disables and triggers a mass status flush, thousands of clients simultaneously increase their polling frequency (due to adaptive polling logic), spiking database connections and hitting Supabase's concurrent connection limit, which freezes all other application functions.  
**Doomsday Scenario**: Complete system failure where even admin interventions are impossible, requiring a Supabase force-restart during peak demand.  
**Technical Solution**: Implement a graduated backoff for polling clients during system-triggered events, using Server-Sent Events or a broadcast mechanism (Supabase Broadcast) to notify all clients to slow their polling. Additionally, use read replicas for queue position queries and implement a connection pooler (PgBouncer) in transaction mode for Supabase.
